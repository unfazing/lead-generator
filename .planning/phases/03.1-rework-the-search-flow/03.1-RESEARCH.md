## Summary

Phase 03.1 should not introduce a new workflow engine or state library. The current code already has the correct durable resources: company recipes, company snapshots, people snapshots, and run plans. The main problem is orchestration. [`src/app/search/page.tsx`](/Users/lian/dev/apollo/src/app/search/page.tsx) currently auto-selects the first company recipe, first people recipe, first company snapshot, and first people snapshot when URL params are absent, then renders every stage on one page. That creates accidental state, hides the real workflow checkpoints, and makes the workspace feel wordier than it is.

The rework should make the active stage explicit in the route and make persisted records, not component nesting, define progress. The recommended flow is: instructions/pairing -> company search -> company snapshot review -> people search setup -> people snapshot review -> run planning. Keep Apollo-facing filter payloads derived from saved recipes and persisted snapshots, because Apollo company search consumes credits while people search is free but still capped at 50,000 displayed rows. That means the product should continue to force review checkpoints before the next paid or broader step.

**Primary recommendation:** Replace implicit first-item defaults and one-page stacking with explicit stage routes whose source of truth is the active recipe/snapshot/run-plan IDs.

### Pattern 1: Stage Route As Source Of Truth
**What:** The active workflow stage must come from the route, not from “first available record” fallback logic.
**When to use:** For switching between instructions, company review, people review, and run planning.
**Example:**
```typescript
// Source: Next.js searchParams/page guidance + local adaptation
type SearchStage = "home" | "company" | "company-review" | "people" | "people-review" | "plan";

type WorkspaceContext = {
  companyRecipeId?: string;
  peopleRecipeId?: string;
  companySnapshotId?: string;
  peopleSnapshotId?: string;
};

export function getStageHref(stage: SearchStage, context: WorkspaceContext) {
  switch (stage) {
    case "home":
      return "/search";
    case "company":
      return `/search/company?${new URLSearchParams(context as Record<string, string>).toString()}`;
    case "company-review":
      return `/search/company/${context.companySnapshotId ?? ""}?${new URLSearchParams({
        companyRecipe: context.companyRecipeId ?? "",
        peopleRecipe: context.peopleRecipeId ?? "",
      }).toString()}`;
    case "people":
      return `/search/people?${new URLSearchParams(context as Record<string, string>).toString()}`;
    case "people-review":
      return `/search/people/${context.peopleSnapshotId ?? ""}`;
    case "plan":
      return `/search/plan/${context.peopleSnapshotId ?? ""}`;
  }
}
```

**What:** Persist decisions that must survive navigation or are operationally meaningful: company snapshot, selected-company scope, people snapshot, run plan.
**When to use:** For anything that gates a later Apollo call or retrieval step.

**What:** Column pickers, expanded params panels, and unsaved row toggles should remain local client state until the user submits a stage transition.
**When to use:** Snapshot review tables and param viewers.
**Example:**

**What:** Do not let the search workspace invent ad hoc payloads. Recipes remain the editable source of filter truth; snapshots preserve exactly what was executed.
**When to use:** Company search requests, people search requests, param viewers, and rerun affordances.
**Example:**

### Anti-Patterns to Avoid
- **Implicit first-item activation:** [`src/app/search/page.tsx`](/Users/lian/dev/apollo/src/app/search/page.tsx) currently falls back to the first recipe and first snapshot. This makes the page look “active” before the user has chosen context.
- **Everything-on-one-page staging:** Company search, people search, people review, and run planning should not all compete for attention in one vertical stack.
- **Client-only workflow state:** Selected companies currently live only inside [`src/features/company-search/components/company-results-workspace.tsx`](/Users/lian/dev/apollo/src/features/company-search/components/company-results-workspace.tsx) until form submit. That is fine for in-table editing, but not as the lasting definition of the people-search input.
- **Instruction-heavy duplicated copy:** Replace explanatory paragraphs with compact stage summaries, badges, and next-step labels.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Workflow stage management | Custom client state machine for the whole search flow | Next route segments + validated params | The browser URL already gives shareable, refresh-safe stage state. |
| Apollo request history display | A second hand-maintained “display params” model | Persisted `recipeParams` on snapshots | Prevents drift between what Apollo received and what the viewer shows. |
| Rerun context tracking | Implicit “latest item” heuristics | Explicit IDs in route and redirect targets | Latest-record inference is what currently causes accidental activation. |
| Cross-stage selection durability | localStorage/sessionStorage staging | Persist selected company scope in the people snapshot | Selection scope affects downstream search behavior and must survive navigation. |

**Key insight:** This phase is a workflow-boundary fix, not a state-management-library problem.

## Common Pitfalls

### Pitfall 1: Treating Search As One Continuous Surface
**What goes wrong:** Users lose the sense of checkpointed progress because the page renders company search, people search, people results, and run planning together.
**Why it happens:** The current search route treats the workflow as one dashboard instead of sequential resources.
**How to avoid:** Split the workflow into explicit stage routes with compact stage nav and only the current step’s primary controls visible.
**Warning signs:** Empty-state cards from later stages render before the user has completed earlier ones.

### Pitfall 2: Letting URL Absence Mean “Pick The First Record”
**What goes wrong:** Refreshing or entering `/search` silently selects recipes and snapshots the user did not intentionally choose.
**Why it happens:** The page falls back to `recipes[0]` and `snapshots[0]`.
**How to avoid:** On missing IDs, show instructions and selection UI instead of auto-selection.
**Warning signs:** A newly-created recipe or snapshot unexpectedly changes the default visible workspace.

### Pitfall 3: Losing The Exact Executed Parameters
**What goes wrong:** Snapshot viewers show recipe summaries, but the user cannot trust that they match the actual Apollo request.
**Why it happens:** The UI can drift toward “saved recipe description” instead of “executed request params”.
**How to avoid:** Always render snapshot param viewers from persisted `recipeParams` and request metadata, not recomputed display summaries.
**Warning signs:** A changed recipe makes an old snapshot look like it used the new filters.

### Pitfall 4: Mixing Ephemeral Review State With Durable Workflow State
**What goes wrong:** Table toggles and column visibility become hard to reason about, or get over-persisted, while real handoff state remains implicit.
**Why it happens:** UI state and workflow state are not clearly separated.
**How to avoid:** Persist only stage outputs; keep table cosmetics local.
**Warning signs:** The code starts storing column-picking or accordion state alongside snapshots and run plans.

### Pitfall 5: Breaking Apollo Contract Alignment During UX Cleanup
**What goes wrong:** A simplified UI stops reflecting Apollo’s real request semantics, causing hidden behavior changes.
**Why it happens:** UX refactors often collapse fields too aggressively.
**How to avoid:** Keep recipe edit pages as the place for filter editing, and keep search-stage viewers tied to the same contract keys Apollo expects.
**Warning signs:** The search workspace begins inventing alternate filter names or silently dropping filters before submission.

## Open Questions

1. **Should company and people recipe pairing be symmetric or company-first?**
   - What we know: the user wants company selection to feed people search, and current actions already treat company snapshot as the upstream artifact.
   - What's unclear: whether the entry page should require selecting both recipes before entering company search, or allow company-first and add people recipe later.
   - Recommendation: plan around company-first with optional people recipe pairing on entry, but require a people recipe before entering the people-search stage.

2. **Should earlier stages remain visible as a sidebar summary while later stages are active?**
   - What we know: the current fully stacked layout feels too wordy.
   - What's unclear: whether the user still wants at-a-glance context from prior stages.
   - Recommendation: keep only a compact stage summary rail, not live editable panels from prior stages.
