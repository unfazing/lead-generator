# Phase 04.1 Plan 01: Contact Batch Enrichment Workflow Summary

**Contact batch persistence, provenance-aware member dedupe, and dedicated enrichment/store routes backed by append-only global Apollo-ID reuse checks**

## Accomplishments
- Added durable `contact-batches` and `contact-batch-members` repositories with CRUD, removal support, and provenance-aware dedupe by Apollo person ID.
- Added dedicated `/enrich` and `/enrich/store` route foundations so enrichment is no longer anchored to the people workflow page.
- Tightened enriched-people persistence and retrieval execution so any Apollo person ID already present in the global store is treated as covered before Apollo calls.

## Decisions Made

- Contact batches remain independent from recipes and are persisted as their own mutable grouping model.
- Batch members preserve snapshot provenance separately from the global store so later member removal does not affect enriched-person outcomes.
- The central enriched-people store now behaves as append-only coverage keyed by Apollo person ID rather than allowing replacement writes.
