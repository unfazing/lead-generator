# Phase 04.1: Contact Batch Enrichment Workflow - Research

**Researched:** 2026-03-27
**Domain:** Enrichment workflow separation, mutable contact batches, and append-only global enriched-person reuse
**Confidence:** HIGH

## User Constraints

### Locked Decisions

- Enrichment should no longer be tightly coupled to the people workflow page.
- Introduce a new logical grouping called a `contact batch`.
- A contact batch is independent from recipes and can contain people drawn from multiple people snapshots.
- Contact batch membership is keyed by Apollo person ID and can be extended over time.
- Adding the same person from multiple snapshots must dedupe batch membership while preserving all provenance references.
- A batch member should store lightweight frozen display data for convenience, including person name, title, company name, and company Apollo ID when available.
- Batch membership should be removable.
- The central enriched-people store is append-only and should not support deletion.
- Before any Apollo enrichment call, the system must check the central enriched-people store by Apollo person ID and skip any person already enriched there.
- This global skip rule applies across all contact batches, including prior unusable outcomes.
- No manual add-by-Apollo-ID flow for v1; batch members come from saved people snapshots only.
- Central enriched people should have their own separate viewer.

### Implementation Implications

- The existing retrieval executor already has the correct global reuse primitive: the central enriched-people store keyed by Apollo person ID.
- What is missing is a stable user-facing grouping above that store; people snapshots are source material, not the long-lived enrichment workspace.
- The new workflow should separate:
  - source selection from people snapshots
  - mutable batch membership and provenance
  - append-only enriched-person outcomes

## Recommended Product Model

### Contact Batches

Contact batches are mutable operator-owned groupings above the global enriched store.

Each batch should contain:

- `id`
- `name`
- `notes`
- `createdAt`
- `updatedAt`

Each member should contain:

- `personApolloId`
- frozen display fields:
  - `fullName`
  - `title`
  - `companyName`
  - `companyApolloId`
- provenance list:
  - `peopleSnapshotId`
  - `sourcePeopleRecipeId`
  - `addedAt`

### Central Enriched People Store

The central store remains the single source of truth for actual Apollo enrichment outcomes.

It should remain:

- append-only
- keyed by Apollo person ID
- reusable across all runs and all contact batches

The UI implication is:

- a batch shows whether each member is already covered by the central store
- running enrichment for a batch only targets the subset missing from the central store

## Recommended Routes

- `/search/people`
  - create and review people snapshots only
- `/enrich`
  - choose a contact batch
  - add/remove people from saved people snapshots
  - enrich selected or all missing members
  - inspect latest run state and batch-scoped enriched results
- `/enrich/store`
  - inspect the append-only central enriched-people store

### `/enrich`

## Data and Execution Rules

### Rule 1: Batch Membership Dedupes By Apollo Person ID

If the same Apollo person is added repeatedly:

- keep one batch member row
- merge provenance refs
- keep the frozen display fields from the most recent add unless empty

### Rule 2: Enrichment Always Checks The Global Store First

When enriching a batch:

- selected members are resolved
- current central enriched-people store is checked by Apollo person ID
- already-covered members are skipped before Apollo calls
- only missing members are passed to the existing retrieval executor

### Rule 3: Prior Unusable Outcomes Still Count As Covered

The user explicitly wants to avoid spending credits twice even for people who previously ended in:

- `no_match`
- `email_unavailable`
- other unusable Apollo outcomes

So the skip rule is:

- any Apollo person ID present in the central enriched-people store is already handled

### Rule 4: Batch Viewer And Central Store Viewer Are Different Concepts

Batch viewer:

- membership + provenance + enrichment state

Central store viewer:

- global enriched person records only

Do not collapse them into one page; operators need both.

## Reuse Patterns From Existing Code

### Reuse Existing Retrieval Executor

Current executor behavior already checks the central enriched-people store before Apollo calls in:

- `src/lib/retrieval/execution.ts`

This should remain the backend enforcement point even after batch extraction.

### Reuse Shared Snapshot Viewer Stack

The existing shared viewer stack can support:

- batch members
- batch-scoped enriched results
- central enriched-people store

Key files:

- `src/features/snapshots/components/data-snapshot-viewer.tsx`
- `src/features/snapshots/components/snapshot-results-table.tsx`

### Keep People Snapshots As Add Sources

The existing people snapshot viewer already supports row selection inside:

- `src/features/people-search/components/saved-people-snapshots-panel.tsx`

That selection pattern can be reused inside `/enrich` as the “add from snapshot” tool.

## Anti-Patterns To Avoid

- Reusing the people workflow as the permanent home of enrichment controls
- Treating one people snapshot as the only owner of enrichment outcomes
- Allowing Apollo re-enrichment for a person already present in the central enriched store
- Coupling contact batches to one people recipe
- Deleting or mutating central enriched-person records

## Summary

Phase 04.1 should introduce a new enrichment layer above people snapshots:

- people snapshots remain source material
- contact batches become the mutable working set
- the central enriched-people store remains the append-only global cache and result store

That gives the operator a durable enrichment workflow without duplicating Apollo spend.
